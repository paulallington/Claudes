# Security Audit ŌĆö 2026-05-06

End-to-end security audit of the Claudes Electron app, run by four parallel
Claude agents covering distinct attack surfaces:

1. Electron IPC + preload boundary
2. pty-server WebSocket (the highest-risk surface)
3. Renderer XSS / DOM injection
4. Main-process file IO + child_process

This branch (`security-audit-2026-05-06`) implements the high-confidence
fixes. Anything requiring user judgement is documented under
"Deferred to user review" below.

## Threat model

The Electron renderer is sandboxed (`contextIsolation: true`,
`nodeIntegration: false`) so the immediate trust boundary is the IPC
surface and the local pty-server WebSocket. The audit looked for paths
where a renderer compromise (XSS, malicious content rendered into DOM,
malicious paste) ŌĆö or a *neighbouring* untrusted process on the same
local machine (a webpage in the user's browser, another local app) ŌĆö
could escalate to:

- Arbitrary code execution as the desktop user
- Reads of unrelated files (ssh keys, credentials, browser cookies)
- Writes that subvert the OS or shared tooling
- Hijacking of live Claude sessions

## Deferred to user review

These are real findings but each requires a judgement call about whether
the proposed fix is the right approach, what the UX should look like, or
how to phase delivery. None is being blocked by this branch ŌĆö the most
important critical issues are already shipped above. Listed in approximate
priority order.

### 1. pty-server: cmd / cwd allow-list

**Finding:** even with the new auth token, pty-server still accepts any
`cmd` and any `cwd`. A renderer compromise ŌĆö or a curious user copy/paste
of a hostile prompt that triggers a tool the renderer obeys ŌĆö could spawn
arbitrary executables. Practically, the renderer only ever spawns
`claude` (and a small set of run-tab launchers the user explicitly
configures), so an allow-list is feasible.

**Why deferred:** the run-tab feature explicitly lets the user run
arbitrary commands (npm start, dotnet run, python, etc.). Implementing
this needs a design choice: either (a) accept any cmd, but reject UNC
paths and require absolute paths to resolve under a known project root, or
(b) keep a per-project list of "approved" commands the user has run
before, plus a one-time confirm dialog for new ones. Want your input
before shipping.

### 2. `endpoint:fetchModels` SSRF

**File:** `main.js:803-857`. Renderer-supplied `baseUrl` is fetched
server-side with optional auth headers. Allows hitting `http://127.0.0.1:*`,
`http://169.254.169.254/...` (cloud metadata), internal services.

**Why deferred:** the user *intentionally* configures the URL via the
endpoint dialog; legitimate use includes `http://localhost:11434/...`
(Ollama) and other local LLMs. A strict private-IP block would break
that. A safer middle ground: keep allow listing public Anthropic endpoints
freely, but require an explicit "Allow local URL" toggle on first save.
Worth a short UX discussion.

### 3. `endpoint:save` baseUrl validation

**File:** `main.js:742`. Same caveat as #2 ŌĆö restricting to `http(s):`
schemes is uncontroversial; the localhost block is the contentious bit.

### 4. Mongo connection-string encryption at rest

**File:** `main.js:3931, 4221` and on disk in `automations.json`. Stored
plaintext; flowed into env for the spawned MCP. Recommend wrapping with
`safeStorage` like the existing `encryptToken` helper, and chmod 0600 on
the temp MCP config file on POSIX.

**Why deferred:** changes the on-disk format, so will need a migration
path for existing automations. Want to plan that with you.

### 5. `automations:import` allow-list

**File:** `main.js:2846`. Even after the path-containment fixes here, an
imported automation can persist `dbConnectionString`, `prompt`,
`skipPermissions: true`, etc. that fire on subsequent runs. Some import
sanitisation already happens (`clonePath: null`, lines 2919/2951) but
`skipPermissions` and `dbConnectionString` survive and should be stripped
or surfaced in the import preview.

### 6. pty-server: reattach ownership

**File:** `pty-server.js` `reattach` case. Any authenticated peer can
reattach to any pty `id`, since `id` is renderer-generated. With the auth
token in place this requires a rogue *first-party* renderer (much higher
bar), but tying ids to the originating connection ŌĆö or having the server
mint and return ids ŌĆö would close the residual concern. Mostly a nice-to-have
once the externally-reachable risk is gone.

### 7. `hooks:configure` consent step

**File:** `main.js:2189`. Rewrites `~/.claude/settings.json` without an
explicit user confirm. Not RCE on its own, but hostile renderer code
could durably wire Claude tool calls through an attacker-controlled local
hook server. Recommend a one-time `dialog.showMessageBox` confirm before
writing the hook config.

### 8. `clipboard:readText` user-gesture gate

**File:** `main.js:2060`. Returns clipboard contents to renderer with no
user gesture required. A renderer compromise can poll for whatever the
user just copied (passwords from a manager). Lower risk for a
first-party app, but worth gating to deliberate user actions.

### 9. `sandbox: true` on BrowserWindow webPreferences

**Files:** `main.js:535, 613`. The preload uses only `contextBridge` and
`ipcRenderer` (no Node modules), so this should be a drop-in. Defense in
depth: even if a future preload accident exposed a Node primitive,
sandbox blocks the renderer from using it. Want manual smoke-test before
shipping in case any platform-specific behaviour breaks.

### 10. Code-signing for NSIS installer

**File:** `release.sh`. electron-updater verifies SHA512 from
`latest.yml` published alongside the asset. If GitHub releases for this
repo are ever compromised, an attacker can ship a malicious update.
Recommend enabling Authenticode signing for the NSIS output (and
documenting the cert handling). Not exploitable today, but worth raising.
